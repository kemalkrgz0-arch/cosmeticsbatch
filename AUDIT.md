# COSMETICSBATCH AUDIT

Audit date: 2026-07-14. Scope: repository, production build, sampled live URLs,
decoder engine, brand data, i18n, SEO, privacy, UX and performance. Statements
about manufacturer formats without an in-repository primary source are marked
`needs verification`.

## Architecture Map

- Next.js 16.2.9 App Router, React 19.2.4, strict TypeScript, Tailwind CSS 4.
- pnpm lockfile/workspace; Node 24 Alpine multi-stage Docker build; standalone
  Next server deployed through `deploy.sh` to Coolify/VPS.
- No database. Brands are a typed table in `src/lib/brands.ts`; decoder rules
  are in `src/lib/decoder/decoders.ts`; checks and photo submissions are
  append-only files in a bind-mounted data directory.
- 331 brand rows; 18 registered decoders; 119 rows have no dedicated decoder.
  No duplicate names/slugs, invalid shelf-life values or unknown decoder ids
  were found by the integrity audit.
- Routes use `src/app/[locale]`; English is prefix-free through `src/proxy.ts`,
  while 43 other active locales use a prefix. English is prebuilt and other
  locales render on demand.
- Brand pages are SSG. Only the 34 brands in `BRAND_DETAILS` are indexable and
  monetizable; other brand tools are `noindex, follow` and excluded from the
  sitemap.
- Metadata, canonical and hreflang are server-generated by `src/lib/seo.ts`.
  Sitemap, robots, JSON-LD, OpenGraph and Twitter metadata use Next metadata
  APIs. The production sitemap sampled during the audit had 78 canonical URLs
  and 3,510 hreflang annotations.
- Decode and photo submission use Node API routes. Decoder implementation stays
  server-side. GA/AdSense/Yandex are build-time configured; Yandex is consent
  gated. AdSense is now loaded only by eligible ad slots.
- Main interactive client bundles are the brand picker, inline result, FAQ,
  theme/language controls, consent, ads and photo submission. The full brand
  table is currently imported by the client-side picker.
- System fonts are used; no build-time Google Fonts request. Two brand-specific
  photos and three generic location photos exist.
- No general unit-test runner was configured at audit start. One script only
  checked that editorial examples returned some date; it did not assert exact
  dates or negative/boundary cases.

## P0 — Critical

### P0.1 Generic fallback can manufacture a date for an unsupported code

- **Evidence:** `checkBatchCode()` always tries `FALLBACK_CHAIN` after the
  selected brand decoder; `FALLBACK_CHAIN` contains the permissive `julian`
  reader. The reader extracts the first 4–6 digit substring rather than proving
  the whole code is a supported format.
- **Files:** `src/lib/decoder/index.ts`, `src/lib/decoder/decoders.ts`.
- **Root cause:** format detection and manufacturer decoding are treated as
  interchangeable evidence.
- **User impact:** mistyped codes, serial numbers and barcode fragments can
  receive a plausible but invented manufacture date.
- **SEO impact:** undermines the core product claim and trust/E-E-A-T.
- **Recommended fix:** do not auto-fallback across manufacturer formats. Return
  unknown unless the brand explicitly opts into a decoder.
- **Regression risk:** medium; previously “decoded” unsupported codes will
  correctly become unknown.

### P0.2 Day 366 is accepted in non-leap years

- **Evidence:** `dateFromDayOfYear(2023, 366)` rolls into 2024-01-01. Callers
  only check `1..366`, so the rollover can be emitted as a valid date.
- **Files:** `src/lib/decoder/decoders.ts` and every Julian-family decoder.
- **Root cause:** JavaScript Date rollover is used without a leap-year guard.
- **User impact:** impossible manufacturing days become confident dates.
- **SEO impact:** incorrect worked examples/results weaken reliability.
- **Recommended fix:** central strict day-of-year validation plus regression
  tests for leap and non-leap years.
- **Regression risk:** low.

### P0.3 Privacy claims contradict production behavior

- **Evidence:** checks are sent to `/api/decode` and human checks are appended
  to JSONL, while `src/app/[locale]/about/page.tsx` and `src/lib/faqs.ts` say
  codes are never stored/logged or are decoded in-browser. Translated feature
  copy also makes broad “private” claims.
- **Files:** `src/lib/dataset.ts`, `src/app/api/decode/route.ts`, About, FAQ,
  message dictionaries and site copy.
- **Root cause:** copy was not updated when decoding moved server-side and the
  quality dataset was introduced.
- **User impact:** users cannot give informed consent/understand retention.
- **SEO impact:** contradictory trust claims are easily verifiable quality
  defects.
- **Recommended fix:** central factual privacy wording; remove absolute claims;
  align all locales before indexing them.
- **Regression risk:** low technically, medium translation/editorial effort.

## P1 — High

### P1.1 Decoder provenance is prose, not enforceable data

- **Evidence:** `Decoder` stores id/label/explanation only. There is no source
  type/reference, verification date, version, limitations or fixture set.
  Several comments say “documented”, “confirmed” or “community-observed” without
  a source reference. Manufacturer claims are **needs verification**.
- **Impact:** confidence cannot be audited or downgraded systematically.
- **Fix:** add non-breaking provenance metadata and classify each decoder as
  VERIFIED, HIGH_CONFIDENCE, ESTIMATED, FORMAT_ONLY or UNKNOWN only after source
  review. Do not infer verification from current confidence.
- **Regression risk:** low for metadata, high if algorithms are changed before
  primary evidence is collected.

### P1.2 Permissive substring regexes accept unrelated identifiers

- **Evidence:** many decoders use `match(/\d{4}/)` or `match(/\d{4,6}/)` with no
  numeric-run boundary. A 12/13-digit UPC/EAN can expose a decodable substring.
- **Files:** Coty, embedded, Julian, Shiseido, P&G, K-beauty, Kenvue, Unilever,
  NAOS and Beiersdorf decoders.
- **Fix:** reject valid long UPC/EAN inputs centrally; then tighten each format
  only when real fixtures prove whether prefixes/suffixes are allowed.
- **Regression risk:** low for checksum-confirmed UPC/EAN rejection; medium for
  blanket regex tightening.

### P1.3 Confidence labels overstate evidence

- **Evidence:** L'Oréal returns `high` although its provenance is not stored;
  year-only and single-year-digit estimates often return `medium`; UI presents a
  single date even when the day is an artificial midpoint.
- **Fix:** separate precision from evidence confidence and expose date precision
  (`day`, `month`, `year`, `unknown`). Provenance changes are **needs
  verification** before reclassification.
- **Regression risk:** medium; API/UI schema changes affect translations and
  result cards.

### P1.4 Estimated expiry is labelled as “Expiration date”

- **Evidence:** result UI and share/calendar actions use “Expiration date” and
  “replace by this date”; engine merely adds a generic shelf-life duration.
- **Files:** `src/lib/decoder/index.ts`, `src/components/result-card.tsx`,
  `src/components/result-actions.tsx`, message dictionaries.
- **Impact:** an estimate can be mistaken for a manufacturer safety deadline.
- **Fix:** label “Estimated unopened shelf-life date”; preserve PAO and printed
  expiry as separate concepts.
- **Regression risk:** low logic, medium translation breadth.

### P1.5 Programmatic pages remain highly repetitive

- **Evidence:** sampled live brand pages shared 59–81% of normalized five-word
  sequences. Only 34 brands have editorial records and only two brand-specific
  images exist. Noindex reduces Search exposure but does not create user value.
- **Fix:** keep unreviewed tools functional and ad-free; promote brands only
  with evidence, real examples, brand-specific locations, limitations and
  reviewed submissions. Do not synthesize “unique” prose.
- **Regression risk:** low when promotion remains allowlist-driven.

### P1.6 Localized routes contain English-only and stale claims

- **Evidence:** legal/static routes and the new photo flow contain hard-coded
  English; locale dictionaries retain old privacy/accuracy claims; English
  fallback can produce mixed-language pages.
- **Fix:** translation completeness validation in CI; noindex a locale/path when
  critical content is missing or publish only reviewed locales.
- **Regression risk:** medium for index coverage.

### P1.7 No error boundaries and incomplete operational visibility

- **Evidence:** no route-level `error.tsx`/`not-found.tsx` exists. Dataset writes
  warn once, while photo writes can surface a generic framework 500. Client
  decode errors are generic and no server correlation id is returned.
- **Fix:** localized safe error boundaries and structured, privacy-safe logs.
- **Regression risk:** low.

## P2 — Medium

- Brand picker ships the entire brand dataset and logo-domain map to the client;
  split the compact picker index from server-only editorial data.
- `check-form.tsx` (352 lines), `result-card.tsx` (294) and large content/data
  modules increase maintenance cost; split by behavior, not aesthetics.
- The custom combobox lacks a labelled combobox input/`aria-activedescendant`
  relationship and virtualization; keyboard movement works but screen-reader
  semantics need a focused audit.
- Result loading/error and photo-flow strings are English-only.
- CSP is absent. Adding one requires nonce-based handling for Next, analytics and
  ads and should not be improvised.
- In-memory rate limits are per process and trust proxy IP headers; enforce a
  second limit at Cloudflare/reverse proxy in multi-instance production.
- Photo submission re-encoding is good for EXIF removal, but server-side image
  decoding/content scanning is **needs verification** before public publication.
- Legal pages are English under every locale URL, creating mixed-language
  alternates.
- Dependency vulnerability status was not proven offline; run an online pnpm
  audit in CI and review lockfile changes rather than auto-fixing majors.

## P3 — Low

- README brand/index counts and deployment target wording are stale.
- A dead static `HOME_FAQ` source duplicates the live message-based FAQ and has
  stale privacy claims; verify import graph before removal.
- Minor hard-coded English metadata and labels remain in localized pages.
- Article `datePublished` currently equals each update date; preserve original
  publication separately when the data becomes available.

## Quick Wins

1. Disable generic decoder fallback and reject checksum-valid UPC/EAN lengths.
2. Strictly reject day 366 in non-leap years.
3. Correct About/static FAQ privacy claims.
4. Replace hard-coded `120+` with a count derived from public brand data.
5. Relabel computed expiry as an estimate in English, then roll through reviewed
   translations.
6. Add deterministic positive/negative decoder regression tests.

## Architectural Issues

- Decoder evidence, format parsing, date precision and product shelf-life are
  only partially separated.
- Brand catalog, picker payload, SEO eligibility and decoder support share one
  large module instead of derived views with explicit invariants.
- Translation publication is presence-based, not review/completeness-based.
- File-backed queues suit one container but need locking/object storage or a
  database before horizontal scaling.

## SEO Issues

- Strong canonical/hreflang/sitemap fundamentals; metadata is server-generated.
- Main risk is content/trust quality, not crawl mechanics.
- Noindex brand pages are still linked heavily and were previously monetized.
- Mixed-language legal/tool UI and unsupported absolute claims weaken trust.
- FAQ DOM and JSON-LD use the same ordered array on active pages; no q/a index
  mismatch was found. The separate unused FAQ constant is stale.

## Decoder Accuracy Risks

- Generic cross-format fallback, non-leap day 366, long numeric identifier
  substring matches and current-month midpoint rejection.
- Single-digit years necessarily have decade ambiguity; current notes acknowledge
  it but results still expose one selected date.
- Provenance for all manufacturer rules is **needs verification**.

## Data Integrity Issues

- Structural audit passed duplicate name, duplicate slug, decoder reference and
  positive shelf-life/PAO checks.
- 119/331 rows intentionally lack a dedicated decoder; they must not receive an
  auto-guessed date.
- Hard-coded `120+` claims drift from the catalog.

## UX / Performance / Accessibility Issues

- Core route is understandable and mobile input sizing avoids iOS zoom.
- Search, recent brands, Enter/Escape and arrow navigation exist.
- The combobox needs stronger ARIA ownership/active-option semantics.
- Estimated vs decoded values are not visually explicit enough.
- Static generation, system fonts and delayed third-party scripts are strengths.
- Full client brand data and multiple client providers are the main bundle audit
  candidates; real CWV needs field/RUM data (**needs verification**).

## Security / Privacy Issues

- Same-origin API checks, input limits, rate limits, non-public photo storage,
  JSON-LD escaping and security headers are present.
- Absolute privacy statements remain inconsistent.
- No CSP; proxy-level rate limiting and online dependency audit are pending.
- Raw batch codes are retained; retention/deletion duration is unspecified.

## Test Coverage Gaps

- No exact expected-date fixtures per decoder.
- No invalid, malformed, case, whitespace, leap-year, rollover, decade,
  barcode or ambiguity assertions.
- No route/API component tests, accessibility automation or metadata snapshots.

## Recommended Implementation Plan

### Phase 1 — correctness and truthfulness

Disable unsafe fallback, add strict calendar/barcode guards, correct privacy and
estimated-expiry wording, introduce deterministic regression tests.

### Phase 2 — evidence model

Add decoder provenance/version/precision metadata without changing algorithms;
review each decoder against primary evidence and real fixtures.

### Phase 3 — data and i18n integrity

Automate catalog invariants, central counts and translation completeness; decide
which locale/path combinations are sufficiently reviewed to index.

### Phase 4 — programmatic content quality

Use reviewed user photos and verified examples to promote brands incrementally.
Keep unverified pages functional, noindex and ad-free.

### Phase 5 — UX, accessibility and performance

Clarify result evidence/precision, upgrade combobox semantics, split picker data,
measure bundle/CWV and then tune design based on field data.

### Phase 6 — operations

Add error boundaries, structured privacy-safe observability, retention tooling,
proxy rate limits, backups and a reviewed publication workflow for submissions.
